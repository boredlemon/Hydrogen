About Nebula

Nebula is a powerful, efficient, lightweight, embeddable scripting language developed by a team at PUC-Rio, the Pontifical Catholic University of Rio de Janeiro in Brazil. Nebula is free software used in many products and projects around the world.

Nebula's official web site provides complete information about Nebula, including an executive summary and updated documentation, especially the reference manual, which may differ slightly from the local copy distributed in this package.

Installing Nebula

Nebula is distributed in source form. You need to build it before using it. Building Nebula should be straightforward because Nebula is implemented in pure ANSI C and compiles unmodified in all known platforms that have an ANSI C compiler. Nebula also compiles unmodified as C++. The instructions given below for building Nebula are for Unix-like platforms, such as Linux and Mac OS X. See also instructions for other systems and customization options.

If you don't have the time or the inclination to compile Nebula yourself, get a binary from NebulaBinaries. Try also NebulaDist, a multi-platform distribution of Nebula that includes batteries.

Building Nebula

In most common Unix-like platforms, simply do "make". Here are the details.

  1. Open a terminal window and move to the top-level directory, which is named nebula-5.4.4. The Makefile there controls both the build process and the installation process.

  2. Do "make". The Makefile will guess your platform and build Nebula for it.

  3. If the guess failed, do "make help" and see if your platform is listed. The platforms currently supported are:

    guess aix bsd c89 freebsd generic linux linux-readline macosx mingw posix solaris

    If your platform is listed, just do "make xxx", where xxx is your platform name.

    If your platform is not listed, try the closest one or posix, generic, c89, in this order.

  4. The compilation takes only a few moments and produces three files in the src directory: nebula (the interpreter), nebulac (the compiler), and libnebula.a (the library).

  5. To check that Nebula has been built correctly, do "make test" after building Nebula. This will run the interpreter and print its version.

If you're running Linux, try "make linux-readline" to build the interactive Nebula interpreter with handy line-editing and history capabilities. If you get compilation errors, make sure you have installed the readline development package (which is probably named libreadline-dev or readline-devel). If you get link errors after that, then try "make linux-readline MYLIBS=-ltermcap".

Installing Nebula

Once you have built Nebula, you may want to install it in an official place in your system. In this case, do "make install". The official place and the way to install files are defined in the Makefile. You'll probably need the right permissions to install files, and so may need to do "sudo make install".

To build and install Nebula in one step, do "make all install", or "make xxx install", where xxx is your platform name.

To install Nebula locally after building it, do "make local". This will create a directory install with subdirectories bin, include, lib, man, share, and install Nebula as listed below. To install Nebula locally, but in some other directory, do "make install INSTALL_TOP=xxx", where xxx is your chosen directory. The installation starts in the src and doc directories, so take care if INSTALL_TOP is not an absolute path.

bin:
nebula nebulac
include:
nebula.h nebulaconf.h nebulalib.h auxlib.h nebula.hpp
lib:
libnebula.a
man/man1:
nebula.1 nebulac.1

These are the only directories you need for development. If you only want to run Nebula programs, you only need the files in bin and man. The files in include and lib are needed for embedding Nebula in C or C++ programs.

Customization

Three kinds of things can be customized by editing a file:

  • Where and how to install Nebula &mnebula; edit Makefile.
  • How to build Nebula &mnebula; edit src/Makefile.
  • Nebula features &mnebula; edit src/nebulaconf.h.

You don't actually need to edit the Makefiles because you may set the relevant variables in the command line when invoking make. Nevertheless, it's probably best to edit and save the Makefiles to record the changes you've made.

On the other hand, if you need to customize some Nebula features, you'll need to edit src/nebulaconf.h before building and installing Nebula. The edited file will be the one installed, and it will be used by any Nebula clients that you build, to ensure consistency. Further customization is available to experts by editing the Nebula sources.

Building Nebula on other systems

If you're not using the usual Unix tools, then the instructions for building Nebula depend on the compiler you use. You'll need to create projects (or whatever your compiler uses) for building the library, the interpreter, and the compiler, as follows:

library:
api.c code.c ctype.c debug.c do.c dump.c function.c garbageCollection.c lexer.c memory.c object.c opcodes.c parser.c state.c string.c table.c tagMethods.c undump.c virtualMachine.c zio.c auxlib.c baselib.c corolib.c dblib.c iolib.c mathlib.c loadlib.c oslib.c strlib.c tablib.c utf8lib.c Initialize.c
interpreter:
library, nebula.c
compiler:
library, nebulac.c

To use Nebula as a library in your own programs, you'll need to know how to create and use libraries with your compiler. Moreover, to dynamically load C libraries for Nebula, you'll need to know how to create dynamic libraries and you'll need to make sure that the Nebula API functions are accessible to those dynamic libraries &mnebula; but don't link the Nebula library into each dynamic library. For Unix, we recommend that the Nebula library be linked statically into the host program and its symbols exported for dynamic linking; src/Makefile does this for the Nebula interpreter. For Windows, we recommend that the Nebula library be a DLL. In all cases, the compiler nebulac should be linked statically.

As mentioned above, you may edit src/nebulaconf.h to customize some features before building Nebula.

Changes since Nebula 5.3

Here are the main changes introduced in Nebula 5.4. The reference manual lists the incompatibilities that had to be introduced.

Main changes

  • new generational mode for garbage collection
  • to-be-closed variables
  • const variables
  • userdata can have multiple user values
  • new implementation for math.random
  • warning system
  • debug information about function arguments and returns
  • new semantics for the integer 'for' loop
  • optional 'init' argument to 'string.gmatch'
  • new functions 'nebula_resetthread' and 'coroutine.close'
  • string-to-number coercions moved to the string library
  • allocation function allowed to fail when shrinking a memory block
  • new format '%p' in 'string.format'
  • utf8 library accepts codepoints up to 2^31
